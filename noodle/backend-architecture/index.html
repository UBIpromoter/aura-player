Aura Assessment Credentials

Architecture & implementation materials for team review

We've been building Aura's assessment engine — 20+ personality assessments producing deterministic scores across Big Five, Shadow, Risk, Attachment, Chronotype, and more. These docs describe how those scores connect to Adam's credential architecture: SMP verification (Mode 1) and BBS+ selective disclosure (Mode 2), wrapped in encrypted sessions.

Two key findings: Foxx can't run BLS12-381 WASM synchronously (sidecar needed for Mode 2), and the architecture phases naturally — Mode 1 delivers most of the spec without any credential infrastructure.

Decisions we need from you

  1. Relay architecture — Nodes as relays? WebRTC direct? Hybrid?
  2. Session layer — Is post-session deniability essential for Phase 1, or can we ship with libsodium and add OTR later?
  3. Blind issuance timing — Day-one for Mode 2, or acceptable as an upgrade from standard issuance?
  4. Team credential scoping — Team-scoped or system-wide via Updraft weighting?
  5. Pseudonym derivation — BrightID-derived (stronger) or separate secret (decoupled)?
  6. Domain scoping — Do personality assessments get their own Updraft campaign?
  7. AI entity uniqueness — Extension of Aura verification, or a fundamentally different domain?
  8. Credential lifecycle — Revocation mechanism? TTL? What happens when a node loses trust?
