Adam's Spec — His Words

Architectural goals for the Aura Player App

Privacy

As users answer questions, they generate local facts about themselves. Some of these facts could be useful to Aura players in evaluating them in domains such as human uniqueness.

We want to help users understand which facts they might want to selectively share.

Those that might be selectively shared can be approved by a user in the following ways:

  • The user approves that anyone can see that a user has a certain fact (e.g. twitter account), but not see the details of the fact (e.g. the twitter account identifier), only check that they already know it. We can offer to combine facts into bundles to discourage guessing or griefing. (E.g. you must check the email address AND the phone number together.)
  • They can choose to share facts directly with players they select.

Goals

  • Share just the facts or parts of facts a user chooses.
  • Plausible deniability: a user proves a fact, but the player (verifier) can't share it further.
  • Allow a player to verify they know approved facts, but they can't learn anything new.

Tools

  • Off the record protocol
  • Socialist millionaire protocol
  • BBS+

Evaluation

Most players will evaluate subjects using information they already know (share nothing architecture).

In some domains, players must corroborate to ensure they're talking about the same subject without publicly revealing information about the subject.

In some domains, (such as proving human uniqueness), players want to know that a subject isn't getting verified multiple times as different subjects using the same proofs. When looking at the shared proofs for a subject, they'll want to ensure that a different subject didn't use some or all of the same proofs to get verified, but this needs to be done in a way that doesn't violate the privacy goals stated above.

Scalability

The architecture must be scalable to billions of participants.

You can assume a working network of BrightID (Aura) nodes and choose to deploy protocols as part of the node software.

Decentralization

No centralized actor should be able to block operations or decrypt private data.

Any protocols used in any of the goals described must conform to this.

Read this article to see how BrightID/Aura/Updraft work together to achieve decentralization.

How We Owned It — v6

Privacy

Adam designed two approval methods. We built to both, using all three tools he specified. The user-facing distinction: "Show a player" vs "Share with a player." Showing = private verification, nothing changes hands. Sharing = handing someone a portable credential. The user's choice of mode IS the privacy decision.

Nailed

"Check that they already know it." Mode 1 (SMP). Four messages, two round trips, neither party transmits the fact. Subject controls participation.

In
Player's existing knowledge + subject's local facts
→
Out
Yes or no. Nothing else. No data transmitted.
Nailed

"Share facts directly with players they select." Mode 2 (BBS+). Each assessment fact = one credential slot. User picks which to reveal. Everything else is mathematically hidden.

In
User's signed credential + their choice of what to reveal
→
Out
Verifiable proof of chosen facts. All others invisible.
Nailed

"Bundle to discourage guessing." Every Aura fact type mapped with entropy analysis. Low-entropy facts (attachment style: 4 values, chronotype: 3) never exposed alone — must be bundled with high-entropy facts (Big Five: ~10 billion combinations). System enforces minimum entropy per bundle.

Nailed

"Plausible deniability." Mode 1: fully deniable. After the session, neither side holds evidence. The verifier knows but can't prove it. Just their word.

Mode 2: not deniable — a signed proof is portable. But that's the whole point of choosing to share rather than show. Your spec already separates these into two methods. The deniability property follows the mode the user picks.

Nailed

"Verify they know approved facts, can't learn anything new." SMP. Zero information leakage beyond the boolean match.

Nailed

Credential signing — blind issuance. The signing node never sees assessment scores. Signs a sealed commitment. This is the only approach where no centralized actor ever sees a user's data, honoring the decentralization rule.

And the fabrication concern doesn't exist. Assessment scores are self-reported. People can already lie on personality tests, have AI coach them, or not even realize their self-perception is inaccurate. Verifying scores at signing time doesn't solve dishonesty. The Aura trust graph is where reliability lives. A credential is only as meaningful as the Aura-verified trust behind the identity that holds it. Over time, through evaluations, the network develops a sense of who's reliable and in what context.

Simpler option

Blind signing is a draft IETF spec, implemented by one library. Standard signing — where the issuer sees the scores — is how every certificate authority in the world works. Well-understood, every library supports it.

Possible path: Start with standard issuance for Mode 2. The privacy cost is limited to the signing moment, mitigated by only allowing high-trust nodes to issue. Upgrade to blind issuance when the spec matures. Adam may already have a strong opinion here — if blind issuance is a day-one principle, we're ready for it. If not, the simpler path gets Mode 2 shipping sooner.

Nailed

Griefing — full picture. Bundling prevents information extraction. Rate limiting + subject veto handle harassment. $1 BrightID sponsorship cost prevents mass spam. Bot armies can inflate their own graph corners but can't fake connections into the verified core. The defenses compound: structural, economic, and social.

Nailed

All three tools used as specified. Encrypted session layer wraps both modes. Adam and Ali have already picked the specific OTR variant.

Simpler option

On the session layer: Existing OTR JavaScript libraries are dead. Custom implementation means constant-time requirements and security audits. Battle-tested alternatives (Noise Protocol, libsodium) provide encrypted sessions with forward secrecy out of the box. What they don't provide is OTR's specific post-session deniability.

If post-session deniability is essential for Mode 1, OTR is the right tool and the custom work is worth it. If the primary need is encrypted forward-secret sessions, a proven library gets there faster and more safely. Deniability can be added as an upgrade. Adam and Ali have almost certainly thought about this — we just don't want to overlook a simpler path.

Evaluation

Nailed

"Share nothing architecture." Mode 1 is primary. No credential infrastructure needed — two parties and a relay. Works before any signing, issuance, or node changes exist. Mode 2 is the opt-in layer on top.

Nailed

"Corroborate without publicly revealing." SMP between players. Both hash canonical subject info. Confirms same person without revealing identifiers. Different domains need different corroboration evidence — maps to the node domain architecture.

Nailed

"Not getting verified multiple times." The gate isn't BrightID creation — anyone can spin up as many as they want. The gate is Aura verification. Getting a BrightID verified through the player network is the hard part. Bot armies can inflate their own corners but can't penetrate the verified core. $1 sponsorship prevents mass spam. The scoring algorithm catches structural anomalies.

For credentials: if issuance requires Aura verification, uniqueness is built in. Additional mechanisms possible but likely not needed.

Future direction

Third-party assessments. Right now assessments are self-reported mirrors. But the most valuable data may be the view from the outside — your spouse's assessment of your personality, your colleagues' read on your work style. The gap between self-perception and external perception is maybe the most interesting signal.

Mode 1 already supports this: if two people take assessments about the same subject, a player can verify whether their views align — without seeing any scores. One outside perspective is an opinion. A hundred start to approach truth. This is where Aura's evaluation system points naturally.

Question for Adam

A persistent AI with memory and ongoing relationships is fundamentally different from an ephemeral agent. Is AI uniqueness an extension of what Aura already does, or a different problem needing its own domain?

Scalability

Nailed

"Scalable to billions." Both modes are private conversations between two parties. No global consensus, no bottleneck. A billion simultaneous verifications don't interfere with each other.

Real bottlenecks at scale: relay capacity (distributable) and the scoring algorithm on a billion-node graph (may need distributed computation). Engineering problems, not architectural limits.

Discovery

"Deploy as node software" — the runtime can't handle it. Foxx (ArangoDB's embedded V8) is synchronous-only. The BBS+ signing library is a multi-megabyte WASM module — likely beyond V8's sync compilation limit.

Proposed: Helper service alongside each node. Node handles graph/scores. Helper handles signing/sessions. One more container in the existing cluster.

Adam's already evolving this infrastructure. He may have a better shape. We framed it as a proposal.

Key insight

The sidecar is only needed for Mode 2. Mode 1 runs client-to-client via relay. No node changes, no new containers, no signing keys. If Mode 1 ships first, the sidecar comes later when it's actually needed.

Decentralization

Nailed

"Can't block operations." Issuance authority from the scoring system — many independent nodes, deterministic. No single node can block. Majority collusion is what the scoring algorithm, random walk auditing, and Updraft public ratification are designed to catch.

Nailed

"Can't decrypt private data." In Mode 1, no data leaves the client. In Mode 2 with blind issuance, the signer sees nothing. Conversations encrypted end-to-end. Relays see metadata only. No point in the chain where a single actor reads a user's data.

Nailed

Decentralization article — fully integrated.

Node domains: Different credential types → different issuer trust sets. Domain-specific, not global.

Scoring: Architecture doesn't couple to a specific algorithm. Changeable without breaking credentials.

Updraft: Prediction market incentives determine per-domain node trust. Early accurate opinions rewarded.

Auditing: Public graph structure (IDChain) + private session content. No conflict.

Teams: Independent energy flows = resilience. One compromised team doesn't poison the rest.

Phased Delivery

The big picture

The architecture phases naturally. Mode 1 delivers most of the spec with none of the hardest infrastructure. BrightID's history has shown that reaching for everything at once — cutting-edge tooling, full decentralization, all features — slows things down. This phasing lets us ship something real while the harder pieces mature.

Phase 1 — Mode 1: "Show a Player"

"Verify they know approved facts, can't learn anything new" ✓

"Plausible deniability" ✓

"Bundle to discourage guessing" ✓

"Most players evaluate using information they already know" ✓

"Corroborate without publicly revealing" ✓

Scalable (pairwise, no global state) ✓

No credentials, no signing, no sidecar, no node changes.

Phase 2 — Mode 2: "Share with a Player"

"Share just the facts or parts of facts a user chooses" ✓

Portable, verifiable credentials ✓

Credential-layer duplicate detection ✓

Adds: BBS+ library, sidecar on nodes, signing key management, credential schemas.

Simpler start: Standard issuance → blind issuance upgrade. Single-signer → threshold upgrade.

By the time Phase 2 ships, the blind signing spec may be finalized, OTR libraries may have improved, and Phase 1 will have provided real production experience with sessions and relay infrastructure. Every cutting-edge choice becomes "use the best available tool when we actually need it" instead of "get everything perfect before anything ships."
